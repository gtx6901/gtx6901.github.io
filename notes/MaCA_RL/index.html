# 一、你现在到底在做什么

先把问题说得非常朴素一点。你现在手里有一个空战仿真环境 MaCA 。在当前这条训练路线里，红方有 10 架战斗机，蓝方由固定规则对手控制，通常是 fix_rule 。环境在每一个时间步都会给红方飞机返回观测，红方策略需要根据这些观测，为每一架飞机选择离散动作。动作执行后，战场状态推进，环境返回新的观测、奖励，以及整局是否结束。你真正希望学到的，不是某一步 “看起来合理” 的动作，而是一套能够在完整对抗局里逐渐提高长期战斗效果的策略。

从自动化专业更熟悉的语言来看，这不是经典意义上 “已知模型下的最优控制” 问题，也不是单一参考轨迹跟踪问题。它更像一个复杂的、部分可观测的、非线性的、带对手博弈的闭环决策问题。你可以把红方策略看成一个控制器，把 MaCA 环境看成被控对象加外部扰动源，但这个对象和我们在控制课上写状态空间方程时不太一样：它的动力学太复杂、太黑箱，环境内部状态不完全可见，对手还会不断改变你面临的局面，而且真正关心的目标不是 “当前位置偏差最小”，而是 “整局战斗的长期效果最好”。强化学习自然就会在这里出现，因为它最擅长处理的恰恰是：在一个你未必能完整建模的动态环境里，通过交互和试错，学习长期更优的决策规则。

所以第一件要建立的心智模型是：你现在做的是 “面向长期收益的闭环策略优化”，而不是 “单步分类”、“固定标签拟合” 或者 “一个静态函数逼近任务”。这个认识一旦建立，后面很多选择就会自然起来。例如为什么我们会关心整条轨迹而不是单个样本，为什么日志里会同时看采样吞吐、策略延迟和 reward 曲线，为什么一个看起来合理的瞬时动作仍然未必是好策略，为什么一次训练曲线的回撤并不自动等于 “模型坏了”，以及为什么 reward 设计会变成整个系统里最敏感、最容易学偏的接口之一。

# 二、从你已经学过的控制和优化出发

强化学习如果脱离你原来学过的知识去理解，会显得像一套自成体系的新宗教：到处是新缩写、新名词、新公式，彼此之间似乎都能联系，但第一次看往往抓不住主轴。其实它的骨架并没有那么陌生。你已经学过状态、输入、输出、反馈；你已经知道系统内部有不可见状态，控制器只能基于观测做闭环；你已经知道最优控制会定义某个长时间区间上的累计目标；你也知道参数优化的基本机器就是损失函数、梯度和迭代更新。强化学习真正增加的，并不是一种完全不同的数学，而是把这些东西放进了一个 “环境未知、观测不全、反馈来自交互而不是人工标签” 的框架里。

比如你在最优控制里见过的目标函数经常写成

J=∑t=0Tℓ(xt,ut)J = \sum_{t=0}^{T} \ell(x_t, u_t) J=t=0∑T​ℓ(xt​,ut​)

它表达的是：我们关心的不是某一个瞬间，而是整个时间区间的累计代价。强化学习里常见的目标写法是

J=E[∑t=0Trt]J = \mathbb{E}\left[\sum_{t=0}^{T} r_t\right] J=E[t=0∑T​rt​]

如果你把奖励 rtr_trt​ 理解为负代价，这两者在精神上是一回事。区别只是：最优控制很多时候假设系统模型已知、约束已知，优化器直接在模型上求；而 RL 更像是在和环境不断交互的过程中，用采样数据去逼近这个长期目标。也就是说，RL 不是不讲最优，而是把 “最优” 的求法从显式模型优化，换成了基于交互数据的统计优化。

再比如你已经知道神经网络训练本质上是在做参数优化。监督学习里，输入和标签通常是现成的，损失函数也直接写得出来；而 RL 里的 “监督信号” 来自环境回报和价值估计，这个信号本身会随着策略的变化而变化。换句话说，监督学习是 “在固定数据分布上优化参数”，而强化学习更像是 “参数在改变，数据分布也跟着改变”。这就是为什么 RL 会比普通监督学习难稳，也为什么你在日志里必须同时关心模型更新和采样过程。

还有一组你之后会频繁看到的概念，其实在控制、统计、估计里都不陌生，那就是偏差与方差。你会看到很多 RL 技巧表面上名字不同，比如 baseline、TD、GAE、V-trace，但本质上它们都在做一件事：在 “估计更准” 与 “估计更稳” 之间找平衡。如果只记这个主轴，很多公式就不会显得支离破碎。

# 三、强化学习的最小定义

如果必须用尽量短的话概括 RL，可以这么说：策略在环境中不断试错，环境通过奖励反馈告诉策略哪些行为在长期上更值得保留，哪些行为应该被压低，于是策略参数逐渐朝向更高长期收益的方向移动。

这一定义里最关键的不是 “试错” 两个字，而是 “长期”。因为如果问题只关心当前一步，那很多事情就会简单很多；真正让 RL 变难的，是当前动作的好坏通常要隔一段时间才能显现出来。在空战问题里，这种延迟尤为明显。一次合理的机动，也许短期内看不出任何直接收益，但它可能会让你在十几步之后进入一个更好的探测角度；一次看起来无害的开火，也许会因为浪费了关键弹药，在几十步之后导致失去整局机会。这就是强化学习和短视贪心策略的区别。

再强调一次，这里没有 “老师” 告诉你每个状态的正确动作是什么。环境只能通过结果反馈来间接告诉你 “这种行为整体上好像有帮助” 或者 “这种行为整体上好像害大于利”。所以强化学习不是做动作标签拟合，而是在做一个带延迟反馈的长期决策优化问题。

# 四、用 MDP 和 POMDP 把问题讲清楚

为了方便分析，RL 通常会先把问题形式化成马尔可夫决策过程，也就是 MDP。它包含状态 sts_tst​、动作 ata_tat​、状态转移 P(st+1∣st,at)P(s_{t+1}\mid s_t,a_t)P(st+1​∣st​,at​)、即时奖励 rtr_trt​ 和折扣因子 γ\gammaγ。所谓马尔可夫性，意思是 “未来只取决于当前状态和当前动作，而不需要额外记住更久远的历史”。这是一个非常强的建模假设，但它给我们提供了一个很好的理论框架。

不过，你的项目严格来说更像 POMDP，而不是 MDP。原因很简单：策略看不到真实完整状态，只能看到观测 oto_tot​。环境内部可能有很多你无法直接感知的信息，例如对方的真实位置、刚刚的轨迹变化、潜在接敌关系等等，而策略实际拿到的只有当前局部图像、一些数值摘要、合法动作 mask 和是否存活之类的信息。因此更准确的说法不是 “策略根据状态行动”，而是 “策略根据观测，或者根据一段观测历史行动”。

这个差别非常重要。因为如果是标准 MDP，单步状态本身就足以做最优决策；而在 POMDP 里，当前观测常常不足以代表决策所需的全部上下文，历史信息就会变得关键。这正是为什么当前工程里 LSTM 不是装饰，而是合理需要：它不是为了显得模型更高级，而是为了让策略能用隐状态去积累一部分 “过去的线索”，弥补当前单帧观测的信息不足。

# 五、轨迹、回报与长期目标

在 RL 中，一局游戏的数据通常被看成一条轨迹：

τ=(o0,a0,r0,o1,a1,r1,… )\tau = (o_0, a_0, r_0, o_1, a_1, r_1, \dots) τ=(o0​,a0​,r0​,o1​,a1​,r1​,…)

训练时真正被拿来优化参数的，并不是孤立的一条状态 - 动作对，而是很多这样的轨迹片段。因为策略真正想优化的是从当前时刻开始的长期累计回报。这个量通常定义为

Gt=rt+γrt+1+γ2rt+2+⋯G_t = r_t + \gamma r_{t+1} + \gamma^2 r_{t+2} + \cdots Gt​=rt​+γrt+1​+γ2rt+2​+⋯

其中 γ\gammaγ 是折扣因子。它的作用不是神秘技巧，而是很朴素：把更远未来的奖励适当打折，同时让无限期的累加在数学上更容易处理。如果 γ\gammaγ 较大，例如当前工程中的 0.995 ，就意味着策略比较重视长期后果，这对空战任务是合理的，因为很多关键收益并不会在当前一步立刻兑现。

真正要记住的是，强化学习中的 “好动作” 不是 “立刻拿到高 reward 的动作”，而是 “能提升从现在开始长期累计回报的动作”。这一点如果不牢牢记住，后面看 reward 设计和日志时就很容易被误导。

# 六、Value、Q 和 Advantage 为什么会反复出现

强化学习里的很多公式看起来难，是因为三个价值概念会来回切换：状态价值 VVV、动作价值 QQQ 和优势函数 AAA。如果你把它们各自的角色搞清楚，后面许多算法都只是这三者之间的不同组合。

状态价值函数

Vπ(s)=Eπ[Gt∣st=s]V^\pi(s) = \mathbb{E}_\pi[G_t \mid s_t=s] Vπ(s)=Eπ​[Gt​∣st​=s]

表示的是：如果我现在处于状态 sss，并且之后一直按照策略 π\piπ 行动，那么从现在开始平均还能拿到多少长期回报。它给你的是 “局面本身值多少钱” 的估计。

动作价值函数

Qπ(s,a)=Eπ[Gt∣st=s,at=a]Q^\pi(s,a) = \mathbb{E}_\pi[G_t \mid s_t=s, a_t=a] Qπ(s,a)=Eπ​[Gt​∣st​=s,at​=a]

表示的是：如果我现在在状态 sss，当前这一步强制选动作 aaa，之后再按策略 π\piπ 往后走，那么平均回报是多少。它比 VVV 更细，因为它评价的是 “局面 + 当前动作”。

优势函数

Aπ(s,a)=Qπ(s,a)−Vπ(s)A^\pi(s,a) = Q^\pi(s,a) - V^\pi(s) Aπ(s,a)=Qπ(s,a)−Vπ(s)

则告诉你：在这个局面里，这个动作相对于 “平均水平” 到底好多少。优势函数之所以重要，是因为策略更新时，我们通常不关心某个动作的绝对价值有多高，而更关心它相对同状态下的其他可选动作是不是更值得增加概率。如果一个动作的 advantage 为正，说明它比平均动作更好；如果为负，则说明它比平均水平更差。这种 “相对评估” 比直接用回报更稳，后面在策略梯度和 PPO 里会成为核心。

# 七、Bellman 方程、TD 和 Bootstrapping 的直觉

Bellman 方程的中心思想非常朴素：长期价值可以递归地写成 “当前奖励 + 下一步剩余价值”。对状态价值来说，可以写成

Vπ(s)=Ea∼π,s′∼P[r(s,a)+γVπ(s′)]V^\pi(s) = \mathbb{E}_{a\sim\pi, s'\sim P}\left[r(s,a) + \gamma V^\pi(s')\right] Vπ(s)=Ea∼π,s′∼P​[r(s,a)+γVπ(s′)]

也就是说，从长期来看，一个状态值多少钱，不需要把所有未来细节一次性展开到底，而是可以递推为当前这一步拿到的回报，加上下一个状态未来还能值多少钱。这种递推思想是强化学习中最基础的骨架之一。

这会自然引出两种估计长期回报的思路。第一种是 Monte Carlo，也就是老老实实把一整局跑完，直接用真实累计回报 GtG_tGt​ 作为监督信号。它的优点是直观、无偏，缺点是方差很大，而且必须等回合结束才能更新。第二种是 Temporal Difference，也就是 TD。TD 的思路是，不等整局结束，就用 “当前奖励 + 下一步价值估计” 来更新当前价值估计。例如最简单的 TD (0) 形式是

V(st)←V(st)+α(rt+γV(st+1)−V(st))V(s_t) \leftarrow V(s_t) + \alpha \big(r_t + \gamma V(s_{t+1}) - V(s_t)\big) V(st​)←V(st​)+α(rt​+γV(st+1​)−V(st​))

中间那项

δt=rt+γV(st+1)−V(st)\delta_t = r_t + \gamma V(s_{t+1}) - V(s_t) δt​=rt​+γV(st+1​)−V(st​)

就是 TD error。它衡量的是 “当前价值估计和一步递推目标之间的差距”。这里出现的一个核心动作叫 bootstrapping，也就是 “用自己的估计来更新自己的估计”。它会引入一些偏差，但换来显著更低的方差和更及时的更新。在长时程问题里，这个折中几乎是不可避免的。

如果把这段话翻译成工程语言，就是：你不可能指望每次都等完整回合结束再学习，也不能只信单步即时奖励，于是你必须依赖一个价值估计器来为远期后果提供代理信号。这就是 critic 存在的理由。

# 八、DQN 到底在做什么，以及它为什么曾经看起来有希望

因为你之前在这个工程里走过 DQN 路线，所以理解 DQN 很重要。不是为了再回去做它，而是为了理解为什么它在某些阶段有吸引力，又为什么在当前问题上越来越吃力。

DQN 的核心思想是学习一个动作价值函数 Q(s,a)Q(s,a)Q(s,a)，并希望它逼近最优动作价值函数 Q∗(s,a)Q^*(s,a)Q∗(s,a)。Q-learning 的 Bellman 最优方程写成

Q∗(s,a)=E[r+γmax⁡a′Q∗(s′,a′)]Q^*(s,a)=\mathbb{E}\left[r+\gamma \max_{a'} Q^*(s',a')\right] Q∗(s,a)=E[r+γa′max​Q∗(s′,a′)]

这句话的意思是，当前动作的最优价值等于当前奖励，加上下一状态里最优动作的折扣价值。DQN 用神经网络去逼近这个 QQQ 函数，并通过 replay buffer、target network、Double DQN、Dueling 之类的工程技巧提高稳定性。在小规模、可完全观测、动作空间不是太大、环境分布相对稳定的问题上，这条路线完全可能是有效的。

它之所以曾经看起来有希望，是因为表面上你的问题也有很多离散动作，也能从环境里不断采样轨迹，还能设计 action mask 来减少无效动作，所以一开始会自然觉得：“那是不是 DQN 也能做？” 答案不是绝对不能，而是 “成本和脆弱性会迅速增大”。这里的关键不是 DQN 的理论对错，而是任务结构和算法结构之间的匹配程度。

# 九、为什么当前问题结构会逐渐把你推向 PPO / APPO

这一点值得反复讲清，因为它是整个路线转向的根本逻辑。你现在面对的不是一个单智能体、短时程、全状态可见、几十个合法动作几乎总是都可用的标准教科书问题，而是一个多智能体、部分可观测、长时程、带动作合法性约束、对手还会影响分布的动态博弈环境。每一个特征单独拿出来都不算致命，但它们叠加在一起时，value-based 路线的脆弱性会迅速放大。

第一，多智能体和固定规则对手会带来非平稳性。即便蓝方不学习，它的存在仍然意味着你面对的是博弈结构，不是静态背景。更重要的是，红方自己的 10 架飞机也在同时影响局面，从单机视角看，其他友机行为也会改变它当前动作的长期后果。于是同样的观测和动作组合，在不同联合局势下，长期结果会更加分散，这对 Q 函数拟合非常不友好。

第二，部分可观测意味着单步观测并不足以刻画完整决策上下文。当前时刻到底该不该追、该不该转向、该不该保留弹药，很多时候并不是看当前一帧局部信息就能决定的，而要结合过去几步的探测历史、敌机相对运动趋势以及自己刚刚执行过的机动序列。这对需要在单个状态上给每个动作都打分的 DQN 来说，会把问题变得更加暧昧。

第三，动作空间很大，而且大部分时间真正合法动作只有其中一小部分。这里总动作数是 336 。对于 DQN 来说，这意味着你要为 336 个动作都估值；很多动作从环境角度根本无意义，但它们仍然会占据网络容量、污染 argmax、影响探索。而 policy-gradient 路线则更容易在分布层面直接把非法动作概率压掉。

第四，空战是一个明显的长链条任务。合理机动、接敌、探测、锁定、进入射程、开火、命中、形成胜局，这中间往往隔着很多步。长时程信用分配本来就是 RL 最难的部分之一，当它再叠加部分可观测和大动作空间时，Q-learning 风格的估计会更加脆弱。

第五，课程式预训练不一定自然迁移。你已经看到过，在 fix_rule_no_att 里学到的 “生存和拿分” 能力，进入真实 fix_rule 后并不会自动变成 “会打仗”。这不是个偶然 bug，而是因为对手能力变化后，任务本身已经变了。value-based 路线在这种分布迁移下更容易失去稳定性。

这些因素合在一起后，策略梯度路线就显得更自然。与其为每个动作都拟合一个稳定的长期 Q 值，不如直接学习 “在当前观测下，什么动作应该更可能出现”。再辅以 critic 提供低方差的 advantage 估计，以及 PPO 对大步更新的限制，这条路在当前工程条件下更符合问题结构。

# 十、策略梯度、Actor-Critic 和 PPO 的连续逻辑

如果把策略梯度路线压缩成一条逻辑链，可以这样理解。最原始的想法是：如果某条轨迹最终回报高，那么这条轨迹中做过的动作，未来应该更容易被选到；如果某条轨迹回报低，那么这些动作的概率应该被压低。这就是 REINFORCE 的出发点。它的经典形式可以写成

∇θJ(θ)=E[∑t∇θlog⁡πθ(at∣ot) Gt]\nabla_\theta J(\theta)=\mathbb{E}\left[\sum_t \nabla_\theta \log \pi_\theta(a_t\mid o_t)\, G_t\right] ∇θ​J(θ)=E[t∑​∇θ​logπθ​(at​∣ot​)Gt​]

这里 πθ(at∣ot)\pi_\theta(a_t\mid o_t)πθ​(at​∣ot​) 是策略输出的动作分布，log⁡π\log \pilogπ 的梯度会告诉你应该怎样调整参数，才能提高或降低这个动作的概率，而 GtG_tGt​ 则提供了 “这次动作整体结果如何” 的权重。

问题在于，直接用 GtG_tGt​ 方差往往非常大。于是人们引入 baseline。只要 baseline 不依赖当前动作，减掉它不会改变期望梯度方向，但会显著降低方差。最常用的 baseline 就是价值函数 VVV。这时候策略梯度里真正起作用的权重，就从 GtG_tGt​ 变成了 advantage，形式上接近

At≈Gt−V(ot)A_t \approx G_t - V(o_t) At​≈Gt​−V(ot​)

这意味着：策略更新时，不再只问 “这次回报高不高”，而是问 “这次动作的结果，是否比这个局面下的平均水平更好”。这一步一旦理解了，Actor-Critic 也就顺理成章了。Actor 负责给出动作分布，Critic 负责估计价值并构造 advantage。Actor 用 Critic 提供的低方差信号来更新，Critic 再通过 TD 或 GAE 逼近长期价值。两者相互配合，使得整个训练比纯 REINFORCE 更稳定。

PPO 则是在这个框架上再加一个关键安全带。策略梯度方法的风险是，一次更新可能把策略改得太猛。由于采样数据来自旧策略，新策略如果和旧策略差太远，优化就会不稳。PPO 的办法很经典：计算新旧策略在相同动作上的概率比值

rt(θ)=πθ(at∣ot)πθold(at∣ot)r_t(\theta)=\frac{\pi_\theta(a_t\mid o_t)}{\pi_{\theta_{\text{old}}}(a_t\mid o_t)} rt​(θ)=πθold​​(at​∣ot​)πθ​(at​∣ot​)​

然后在目标函数里对它做裁剪，得到 clipped objective：

LCLIP(θ)=E[min⁡(rt(θ)At, clip(rt(θ),1−ϵ,1+ϵ)At)]L^{CLIP}(\theta)=\mathbb{E}\left[\min\left(r_t(\theta)A_t,\ \text{clip}(r_t(\theta),1-\epsilon,1+\epsilon)A_t\right)\right] LCLIP(θ)=E[min(rt​(θ)At​, clip(rt​(θ),1−ϵ,1+ϵ)At​)]

它的直觉并不复杂：如果更新还在合理范围内，就允许它正常推动策略；一旦某个动作的概率变化太大，就把继续增益截住，防止策略一脚油门冲出稳定区间。当前工程正式训练中真正依赖的稳定性，很大一部分就来自这一思想。

# 十一、GAE、APPO 和 V-trace 应该如何连起来理解

到了这里，常见的困惑是：既然已经有 PPO 了，为什么还会出现 GAE 、 APPO 、 V-trace 、 max_policy_lag 这些看起来像 “另一套东西” 的名词？答案是它们其实不是另一套东西，而是在解决相邻层次的问题。

GAE，也就是 Generalized Advantage Estimation，解决的是 advantage 怎么估。前面已经说了，advantage 是策略更新的重要权重，但它既不能太噪，又不能太偏。GAE 的做法是把 TD residual 进行衰减累加。先定义

δt=rt+γV(ot+1)−V(ot)\delta_t = r_t + \gamma V(o_{t+1}) - V(o_t) δt​=rt​+γV(ot+1​)−V(ot​)

然后构造

AtGAE(γ,λ)=δt+γλδt+1+(γλ)2δt+2+⋯A_t^{GAE(\gamma,\lambda)} = \delta_t + \gamma\lambda \delta_{t+1} + (\gamma\lambda)^2 \delta_{t+2} + \cdots AtGAE(γ,λ)​=δt​+γλδt+1​+(γλ)2δt+2​+⋯

这里的 λ\lambdaλ 就是调偏差与方差平衡的旋钮。你不需要第一次就把推导背下来，只要记住它在做的事：它用很多步 TD 信息，构造一个比单步 TD 更有远见、比完整 Monte Carlo 更稳的 advantage 估计。

APPO 则是在训练架构层面引入异步并行采样。因为环境交互很慢，如果让 learner 既采样又更新，吞吐会很差。于是实际工程里会有多个 actor worker 并行采样，learner 独立更新参数。这大幅提高了吞吐，但带来一个副作用：采样时使用的策略版本可能比 learner 当前训练的版本旧。这就是 policy lag。

V-trace 正是在这个场景里出现的。它是一种重要性采样修正方法，用来缓和 “样本来自旧策略” 带来的偏差。你可以先把它理解成：当异步采样不可避免时，它帮助我们别让过时样本把训练方向带歪。与之配套的 max_policy_lag 则是更工程化的一层保护：如果某些样本太旧，就干脆限制它们的参与范围。当前工程里最近把它从更松的配置收紧到 15 ，核心原因不是因为 “理论上 15 更漂亮”，而是最近真实实验里 learner 积压明显，policy lag 需要更积极地控制。

所以，如果用一句话串起来：GAE 解决 advantage 怎么估，PPO 解决策略怎么稳地更，APPO 解决吞吐怎么提，V-trace 和 max_policy_lag 解决异步采样带来的旧样本偏差。

# 十二、为什么当前工程必须使用 LSTM

只要把 “POMDP” 三个字真正接受，LSTM 就不再像额外附加的黑盒，而更像一个合理的工程后果。因为当前策略看到的只是局部观测，很多关键决策都依赖一小段时间内的上下文。例如目标是否刚刚出现、刚刚消失、自己是否已经转入某个扇区、上一拍的相对角度变化是否意味着对手正在逃逸，诸如此类的信息并不会完整地编码进单帧观测里。

LSTM 的作用不是 “让网络更深”，而是提供一个可学习的隐状态记忆。你可以把它粗略理解成：每一步除了显式观测 oto_tot​，策略内部还维护着一个隐变量 hth_tht​，它由过去观测和过去动作的压缩信息组成。于是策略实际做决策时依赖的是

π(at∣ot,ht)\pi(a_t \mid o_t, h_t) π(at​∣ot​,ht​)

而不是只依赖单一的 oto_tot​。

当前工程里正式基线采用 hidden_size = 256 。这不是一个从理论严格推出来的数字，而是长期受硬件约束、吞吐压力和模型能力之间折中的结果。太小可能记不住关键上下文，太大则会加重 learner 负担，放大异步训练的积压风险。最近工程也证明了这一点：模型不是越大越强，关键是当前数据吞吐和更新速度能否支撑住。

# 十三、动作 mask 为什么在这个项目里是核心组成，而不是小优化

在一些简单环境里，动作 mask 可以只是锦上添花。但在当前问题里，它几乎是核心必需件。原因很直接：总动作数有 336 ，而很多时间步真正合法的动作只有其中很少一部分。如果策略每一步都在完整 336 维分布上自由分配概率，它会把大量概率浪费在环境根本不接受或者根本无意义的动作上。这不只是 “探索有点浪费”，而是会直接占用策略容量、污染熵项、干扰优势更新，最后让训练变得很不经济。

当前工程里动作 mask 的生成逻辑在 fighter_action_utils.py 。它根据当前信息向量里的目标 id、目标距离、长程和短程导弹剩余数量，构造出长度为 336 的合法动作 mask。这个 mask 随时间变化，是动态的。之后，在 scripts/train_sf_maca.py 里，策略前向会把这个 mask 真正作用到 logits 上，把非法动作对应的 logit 压成一个极小值 -1e9 。这样 softmax 以后，非法动作概率几乎为零。

这里有一个非常值得记住的工程原则：环境层兜底和策略层 masking 是两回事。环境层兜底只能保证 “就算策略输出了坏动作，系统也别崩”，它不能直接改变策略梯度。而策略层 logits mask 则是真正把 “非法动作不该被考虑” 这个知识注入到分布里。因此当前工程同时保留了两层机制：策略层负责真正学习时不去碰非法动作，环境层负责在极端情况下依然有安全 fallback，不让环境炸掉。这是经验换来的双保险。

# 十四、奖励设计为什么会成为训练成败的分水岭

很多初学者会下意识地把注意力全部放到算法名字和网络结构上，仿佛只要用了 PPO 、 LSTM 和足够大的 batch，模型就应该自己学会目标。事实恰恰相反。强化学习里最常见的失败，不是因为算法 “太弱”，而是因为你通过奖励告诉模型去学的事情，本身就和你真正想要的目标不完全一致。

奖励不是事实，而是接口。它是你和 agent 之间的沟通语言。你真正想说的是 “去赢得战斗”，但你没法直接把这一整句话塞进网络里，于是只能拆成一些数字：击中给分，被击毁扣分，赢局给分，平局怎么处理，是否鼓励开火，是否惩罚无效动作，是否有每步存活代价，等等。问题在于，一旦某个 shaping reward 的权重失衡，模型就可能开始钻空子，学会了一个在 reward 上看起来很对，但在真实目标上很歪的行为模式。

当前工程已经做过一轮重要修正。过去存在一个显著风险：合法开火动作本身奖励过大，会诱导策略学会 “只要能打就打”，而不是学会 “为了赢局在合适时机使用火力”。因此现在 reward_strike_act_valid 已经调到 0 。这不是说开火就不该被鼓励，而是说 “值不值得开火” 应当主要由命中收益、打空代价、整局输赢后果共同决定，而不是靠一个单独的动作存在奖去驱动。与此同时， reward_draw 也从一个比普通失败更糟的值修正到了 -500 ，使得平局与失败之间的相对关系更合理。

另一个必须一起看的地方是 reward scale 和 clip。当前默认是 reward_scale = 0.005 、 reward_clip = 30.0 。这组参数并不是随便写的。它们共同决定了：整局赢局、完全胜利、局部击毁、每步代价这些信号进入优化器前的相对量级。如果 scale 太大、clip 太小，普通赢局和完全胜利可能都会被裁到相近的上限，critic 就会失去区分能力；如果 scale 太小，又会让训练信号过弱。因此理解 reward 设计时，不要单看环境原始 reward 数值，而要看 “原始奖励 — 缩放 — 裁剪” 这三层之后的最终训练信号。

# 十五、把当前工程看成一条连续的数据流

很多时候文档读着不顺，是因为代码被按文件讲，而不是按数据怎么流讲。现在我们换一种顺序。假设训练中的某一个时间步开始了，发生的事情是这样的。

首先， marl_env/maca_parallel_env.py 这一层会从原始 MaCA 环境拿到红方与蓝方观测，然后只暴露红方战斗机作为学习 agent。它为每个红方飞机构造一个 dict，其中包含图像观测 screen 、数值向量 info 、合法动作 mask 和 alive 状态。如果飞机已经死亡，它不会直接把这个 agent 从系统里删掉，而是保留固定槽位，同时把观测清零，并让只有 no-op 动作合法。这样做的目的，是让多智能体 RL 框架在一整局中始终面对固定 agent 集合，避免中途 agent 数变化带来的复杂性。

接着， marl_env/sample_factory_env.py 会把这些观察进一步整理成 Sample Factory 需要的形式。图像从 HWC 调整成 CHW ，向量观测被归一化后存成 measurements ，同时暴露 action_mask 和 is_alive 。环境步进时，如果策略传来的动作仍然非法，这一层会调用 _sanitize_action 做安全修正，并统计这一整局的非法动作比例。回合结束时，这一层还会把 true_reward 、 round_reward 、 opponent_round_reward 、 episode_len 、 win_flag 等额外统计信息打进日志接口里。

然后策略前向开始。 marl_env/sample_factory_model.py 中的 MaCAEncoder 会先用卷积网络处理图像 obs ，再用一个两层 MLP 处理 measurements + is_alive ，最后把两部分特征拼接起来。 Sample Factory 的 actor-critic 主体会在这个编码结果之上，再通过共享 backbone 与后续 head 生成 value 估计和动作 logits。

接下来，在 scripts/train_sf_maca.py 里，我们对 Sample Factory 自带的 actor-critic forward 做了 monkey patch，把 action_mask 真正作用到 logits 上。也就是说，当前时刻环境判定非法的动作，在策略分布里也会被强制压到几乎零概率。这样采样出来的动作才真正和环境规则一致，而不是依赖后续兜底修正。

策略采样动作后，环境推进一步，得到新的观测和 reward。actor worker 把这一整步的数据，包括观测、动作、log-prob、value 预测、reward、done 等，写入共享轨迹缓冲区。积累到足够长度的 rollout 后，learner 读取这些轨迹，计算 advantage 与 value target，并执行 PPO / APPO 风格更新。更新结束后，新的参数版本再广播回 policy worker，供下一轮采样使用。

这样，整个系统就形成了一条完整的循环：环境产生观测，策略在 mask 约束下采样动作，环境推进并给出 reward，轨迹被收集，learner 更新参数，再把新参数送回采样端。表面上看这只是工程流水线，实际上它正是当前这套强化学习系统的 “物理过程”。

# 十六、当前代码中每一层模块的职责

虽然我更希望你按数据流理解系统，但文件职责还是要有清晰地图。当前工程里，和 RL 主线最相关的代码入口可以按下面的连续逻辑来理解。

marl_env/maca_parallel_env.py 是多智能体环境标准化的核心。它把原始 MaCA 环境包装成更接近 PettingZoo / Gymnasium 并行多智能体接口的形式。它负责定义 agent 集、构造观测、生成合法动作 mask、处理死机槽位、合成 per-agent reward、暴露全局状态信息。

marl_env/sample_factory_env.py 是面向 Sample Factory 1.x 的适配层。它不是策略本身，而是 “让框架能吃这个环境” 的桥梁。这里面最值得关注的是观测格式转换、measurements 归一化、非法动作兜底和 episode 统计写入。

marl_env/sample_factory_model.py 是自定义 encoder，它回答 “这份观测应该怎样编码”。如果以后你想尝试更强的感知结构，这里会是很核心的入口。

marl_env/sample_factory_registration.py 则相当于 “把环境和默认超参数正式注册进框架” 的地方。当前工程中最重要的一批默认值，例如 hidden_size=256 、 rollout=32 、 recurrence=32 、 num_workers=4 、 learning_rate=2e-5 、 reward_scale=0.005 、 reward_clip=30.0 、 ppo_epochs=4 、 max_policy_lag=15 ，都集中在这里。

scripts/train_sf_maca.py 是实际训练入口。它不只是简单转调 Sample Factory ，还承担了几个非常关键的兼容与定制任务，包括 checkpoint 临时文件保存补丁、单轨迹 buffer 形状兼容补丁，以及策略层 action masking 补丁。换句话说，它是 “原框架” 和 “当前项目需求” 之间真正完成结合的地方。

scripts/run_sf_maca_gpu_smoke.sh 用来验证训练链路能否最小闭环，而 scripts/run_sf_maca_4060_baseline.sh 则是正式基线训练脚本。前者是排障工具，后者才是真正拿来做长时间实验的入口。

最后， scripts/eval_sf_maca.py 则是为了补上训练闭环中的 “评估” 这一环。它会加载已有 experiment 的配置和最新 checkpoint，在固定对手和固定局数下运行评估，并给出 win_rate 、 round_reward_mean 、 true_reward_mean 、 invalid_action_frac_mean 等摘要。理解这一层很重要，因为没有独立评估，训练就只剩 reward 曲线猜谜。

# 十七、当前正式基线超参数到底在表达什么

如果参数只是被记成一串默认值，那你每次调参都会很被动。更好的方式是给每个参数一个明确的语言解释。当前正式基线里， num_workers = 4 表达的是采样吞吐和 learner 承载能力之间的平衡。之前更高 worker 数时，真实日志中已经反复出现 learner 积压严重的现象，所以当前收到了 4。它不是绝对正确，只是当前硬件和模型规模下更稳的折中。

rollout = 32 和 recurrence = 32 则一起决定了时序信用分配窗口和 RNN 反向传播长度。rollout 太短，长期动作链会被切碎；太长，则会加重 batch 准备和 learner 压力。当前的 32 是在单卡上兼顾时序信息和吞吐的折中。 batch_size = 512 同样属于这种平衡。太小梯度噪声大，太大又会让更新成本太高，增加异步训练中的陈旧样本问题。

ppo_epochs = 4 表示每批样本会被重复利用多次，而不是只更新一遍就扔掉。这比之前更少 epoch 的设置样本效率高，但又不至于像过多 epoch 那样在旧样本上反复过拟合。 learning_rate = 2e-5 则反映了最近实际训练经验：当策略已经学出一些有价值行为后，更怕大步更新把它冲坏，因此把学习率从更激进的数值收低。

gamma = 0.995 表达的是 “任务更偏长期”。 exploration_loss_coeff = 0.01 则是熵正则强度，防止策略过早塌成接近确定性的分布。 max_policy_lag = 15 体现的是对异步训练中样本新鲜度的更严格控制。最后， reward_scale = 0.005 与 reward_clip = 30.0 则共同控制训练信号进入优化器前的量级层次，必须一起理解。

如果你能把每个超参数都解释成一句 “它在控制什么风险”，那么你后面就会越来越像是在做工程诊断，而不是拍脑袋调数字。

# 十八、如何读训练日志，而不是只看热闹

训练日志最容易让人产生错觉。很多人一看到某个 reward 峰值就兴奋，一看到某段回撤就恐慌，实际上 RL 日志应该分层阅读。

第一层是系统是否正常启动。环境是否初始化成功，learner 是否成功起，policy worker 是否成功起，checkpoint 是否能保存。这一层如果不过，根本不值得讨论算法行为。第二层是设备是否真的在按预期使用。当前工程里，单独那句 Queried available GPUs: 0 并不能直接说明 learner 没用 GPU，你真正要看的是 learner 与 inference worker 是否设置了 CUDA_VISIBLE_DEVICES ，是否打印出 Visible devices: 1 ，以及后续是否出现 GPU learner timing 这样的统计。只有这些组合在一起，才能确认 GPU 真在参与训练。

第三层是吞吐与 learner 负载。当前工程里有一个特别值得警惕的告警： Learner ... accumulated too much experience 。这说明 actor 采样速度已经明显快于 learner 消化速度，样本会越来越旧，异步训练里的 policy lag 风险随之提高。这个问题往往比 “某一段 reward 没涨” 更加基础，因为它会从底层破坏训练稳定性。最近一轮参数调整中，把 num_workers 从 6 降到 4、把 max_policy_lag 收紧到 15，本质上都是在处理这类问题。

第四层才是任务表现本身。这里不能只看单点 reward，而应结合滑动平均、最佳窗口、最后阶段表现、checkpoint 保存时刻和独立评估结果一起看。你需要养成一种习惯：区分 “某一瞬间碰到了好结果” 和 “策略整体进入了更好的统计区域”。这也是为什么固定 checkpoint 的独立评估是必要的，因为训练在线日志本来就更嘈杂。

# 十九、为什么训练 reward 不是最终战斗能力

这一点非常容易在实际实验中被忘掉。训练 reward 是由很多项叠加出来的，它只是一个近似训练目标，不是最终战斗能力的直接定义。即便 reward 在上涨，也可能只是因为策略更会存活、更会拖时间、或更会利用局部 shaping 奖励，而不一定代表它更会赢。

另一个现实原因是，训练过程中对手通常固定为某个规则策略，例如 fix_rule 。这意味着即使训练 reward 稳定上升，也有可能是在过拟合这个固定对手，而不是形成了更强的一般战斗能力。要判断是否真的变强，最靠谱的方法仍然是：固定 checkpoint、固定对手、固定局数，用相同条件重新评估，然后比较 win rate 和整局统计。

此外，PPO / APPO 的训练曲线本身就不应该被期待成监督学习里那种单调收敛的漂亮图。波动、回撤、恢复、本地最优窗口、阶段性退化，在 RL 中都很常见。真正重要的不是 “有没有波动”，而是 “长期趋势是什么、最佳窗口在哪里、最后留下来的 checkpoint 质量如何、独立评估是不是变好了”。

# 二十、Resume、Fresh Start 和 experiment 生命周期

最近工程里真实踩过的几个坑，值得写进主文档，因为它们不是表面小事，而是 RL 工程里非常典型的 “训练管理知识”。

首先， Sample Factory 是用 experiment 名字识别实验目录的。只要 EXP_NAME 不同，它就一定新建目录，从头开始。因此 resume 的第一个前提就是 experiment 名字完全一致。其次，如果某个 experiment 已经达到了设定训练终点，目录里会留下一个 done 文件。此时即使 experiment 名字相同，再次启动也会立刻退出，并提示移除 done 文件才能继续。因此 resume 的第二个前提是：如果你真要继续训练，必须确认 done 文件处理正确。

更容易被忽略的是 train_for_env_steps 。恢复训练时，旧 checkpoint 里已经记录了当前训练迭代数和环境步数。如果旧 checkpoint 的 env_steps 已经是 5000 万级别，而你新命令里的 train_for_env_steps 还停留在旧的上限，那训练一恢复就会发现 “目标步数已经到了”，于是几乎立即结束。也就是说，resume 不只是 “接上旧模型”，还必须保证新的步数上限高于旧 checkpoint 中记录的当前步数。这一点最近已经在实际训练中反复验证过。

什么时候该 fresh start 呢？一般只有在模型结构、观测结构、动作定义发生实质变化，或者你明确想做从零对照实验时，才值得这么做。像当前这种主要调整稳定性超参数、checkpoint 保留策略和采样负载分配的场景，本质上是 “在已有基础上稳定细化”，续训往往更合理。

# 二十一、为什么评估脚本不是附属品，而是闭环的一半

如果没有评估脚本，训练在实践中会退化成一种很低效的行为：盯着 reward 曲线猜模型是不是变好了。当前工程里已经专门补了 scripts/eval_sf_maca.py ，目的就是把 “猜” 尽量变成 “测”。它会加载 experiment 配置和最新 checkpoint，在固定对手、固定最大步数和固定局数下运行评估，然后输出 win_rate 、 round_reward_mean 、 true_reward_mean 、 invalid_action_frac_mean 等摘要指标。

之所以要强调它，不是因为这只是一个方便的小工具，而是因为 RL 里训练数据分布会随着策略改变，在线 reward 本身就不是稳定测量条件下得到的。真正值得比较的，是相同评估条件下不同 checkpoint 的表现。只要你开始用独立评估来决定 “哪个 checkpoint 值得继续训、哪个值得回滚、哪个值得长期保存”，你的实验管理水平就会上一个台阶。

如果把训练比作 “生成候选策略”，那评估就是 “在统一考场里考试”。没有统一考场，你就很难知道某个 reward 峰值到底是偶然、局部过拟合，还是策略真的更强。

# 二十二、当前技术路线真正想做成什么

当你把上面所有内容串起来后，会发现当前工程的主线其实很清楚。第一步不是直接追求 “最强空战策略”，而是先把环境整理成一个现代 RL 框架能可靠接入、能并行采样、能保留动作合法性约束、能输出可用统计指标的标准化训练对象。第二步不是盲跑长训，而是先通过 smoke test 证明：环境、模型、learner、checkpoint 这些系统工程部件都能形成真实闭环。第三步才是在有限 GPU 条件下建立一个真正可持续运行的 baseline，也就是当前这条 Sample Factory + APPO/PPO + LSTM + 共享策略 + policy-level action masking 路线。第四步是建立评估闭环，不再只靠训练日志猜进展。第五步才是更高阶增强，例如 opponent pool、自博弈、更强 critic 或更复杂 curriculum。

这条主线的价值在于，它把强化学习从一种 “靠运气出结果” 的实验活动，逐步变成一个可以解释、可以诊断、可以累积经验的工程过程。你未来就算换环境、换任务、换算法，只要真正理解了这条主线，都会更快进入状态。

# 二十三、如果你接下来要怎么读

为了让阅读更连贯，我建议你不要把这份文档当词典翻，而是分三遍读。第一遍从头顺读到 “当前技术路线真正想做成什么”，先建立整条故事线，尤其把 “这是一个怎样的问题” 和 “为什么会走到 PPO / APPO 路线” 这两件事吃透。第二遍回到中间，重点琢磨 Value、Advantage、TD、GAE、PPO clip、V-trace 这几段数学与直觉的衔接，不追求一次记住所有公式，但要知道每个对象在解决什么问题。第三遍再打开代码，对照 marl_env/maca_parallel_env.py 、 marl_env/sample_factory_env.py 、 marl_env/sample_factory_model.py 、 marl_env/sample_factory_registration.py 、 scripts/train_sf_maca.py 、 scripts/eval_sf_maca.py 去看系统是如何真正落地的。

如果你按这个顺序读，阅读体验会比一开始把所有文件和所有名词同时摊开好很多，因为你的脑中会始终有一条主线：我不是在背零散知识点，而是在理解一套真实系统为什么会长成现在这样。

# 二十四、最后用一句话收束

当前这套工程之所以值得认真学，不是因为它 “恰好能跑”，而是因为它把一个典型但困难的强化学习问题，完整地呈现成了一套从建模、算法选择、观测设计、动作约束、奖励修正、训练稳定性控制、日志诊断到评估闭环的连续工程过程。你真正要学会的，不只是 PPO 这个名字，也不只是某条命令怎么写，而是这种把复杂动态决策问题一步步变成可训练、可解释、可迭代系统的能力。
